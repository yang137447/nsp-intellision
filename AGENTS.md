# AGENTS.md

本文件面向在本仓库中执行任务的 AI 协作者。目标是让代码、测试、资源和文档始终保持一致。

## 先读什么

开始任何任务前，优先阅读这些文件：

1. `README.md`
2. `docs/architecture.md`
3. `docs/resources.md`
4. `docs/testing.md`

如果任务明确涉及某个专题，再补读相关执行方案、专题文档或 human-ai 协作设计稿：

- `docs/` 下与当前任务直接相关的专题文档
- `docs/human-ai/` 下与当前任务直接相关的协作设计稿或共享 skill
  - 该目录用于沉淀人类与 AI 协作过程中的背景、方案、权衡、待确认事项和共享 skill
  - 默认不属于当前事实文档，除非某份文档明确写明自己已升格为事实
- `docs/client-editor-features.md`
  - 当任务涉及注释切换、自动配对、`wordPattern`、snippets、folding、语言扩展名归属等 client 编辑器壳层能力时优先阅读
- `docs/type-method-interface-contract.md`
  - 当任务涉及 `type_model.*`、`methods/object_methods`、array texture 坐标规则，或 `label: Type name` / `label: expr` 一类参数标签语法时优先阅读

默认情况下，`README.md` 和 `docs/{architecture,resources,testing}.md` 才是当前事实文档；当任务涉及对象类型 / 对象方法共享契约时，`docs/type-method-interface-contract.md` 也属于当前事实文档。

## 仓库地图

- `client/`: VS Code 扩展客户端
- `server_cpp/`: C++ LSP 服务端
- `server_cpp/resources/`: 运行时资源 bundle
- `src/test/`: VS Code 集成测试入口
- `test_files/`: 测试夹具
- `scripts/`: 资源脚本、门禁脚本、验证脚本
- `docs/`: 仓库文档目录，包含当前事实文档与协作设计稿
- `docs/human-ai/`: 人类与 AI 协作区域，用于沉淀设计稿、任务背景、方案权衡与共享 skill；默认不作为当前事实来源

## 当前事实规则

- 只支持 bundle 资源布局，不要重新引入 flat 文件命名。
- bundle 统一约定：
  - `base.json`
  - `override.json`
  - `schema.json`
- 资源路径变更后，必须同步更新文档和脚本引用。
- 当前关键共享入口：
  - `server_cpp/src/resource_registry.*`
  - `server_cpp/src/language_registry.*`
  - `server_cpp/src/hlsl_builtin_docs.*`
  - `server_cpp/src/type_model.*`
  - `server_cpp/src/server_request_handlers.cpp`

## 默认执行模式

- 默认把任务理解为：在现有架构内做最小、正确、可验证的改动，不主动扩展为“顺手优化 / 顺手重构 / 顺手补健壮性”。
- 先定位根因，再修改代码；如果根因尚不清楚，先停在诊断和证据，不要用兜底逻辑掩盖症状。
- 一次只解决当前用户明确提出的问题；发现额外问题时，只记录，不顺手修复。
- 优先复用现有共享模块、registry 和单一事实来源，不要在 feature 代码里复制一份规则、表或分支。
- 优先选择改动面最小的方案；不要为了“看起来更通用”而引入额外抽象层。

## 连续执行与上下文收敛

- 当用户已经给出明确执行方案，且任务不触发“必须停下来确认的情况”，也不存在真实阻塞时，AI 应默认连续执行到当前任务完成；中途同步进度可以有，但不应把阶段性里程碑当成新的等待点。
- 对已明确授权连续推进的任务，不得因为“做完一个子阶段 / 一个 milestone / 一轮修改”就主动停下等待用户再输入“继续执行”；只有在触发必须确认项、与用户现有改动直接冲突、或缺少无法安全假设的关键信息时，才允许暂停。
- 需要长链路连续执行时，AI 应主动关注上下文窗口消耗；当估计当前线程上下文已接近 80% 时，必须优先做一次可续接的进度沉淀，而不是把关键信息继续只留在对话里。
- 这类进度沉淀默认优先写入与当前任务直接相关的现有 Markdown；如果用户已经提供执行方案、设计稿或专题文档，应优先在该文档中补充当前进展。
- 如果当前任务没有现成的任务 Markdown，可在 `docs/human-ai/` 下新增一个与任务直接相关的 progress / handoff 文档，至少写清：已完成事项、当前结论、剩余步骤、风险点、待确认事项，以及新线程续接所需的最小上下文。
- 完成上述沉淀后，应明确提醒用户可以新开线程，并指向刚刚更新的 Markdown 作为续接入口；不要等到上下文耗尽后再被动中断。
- 这类 progress / handoff Markdown 默认属于协作沉淀，不自动视为当前事实文档；除非文档明确声明自己已升格为事实，否则后续任务仍以既定事实文档为准。

## AI 可维护性约束

- AI 在新增或拆分模块/类时，应优先保持单一职责；一个模块/类只承载一个主要责任，不把不直接相关的编排、缓存、资源加载、渲染、兜底逻辑混在一起。
- 对已有历史大文件，默认不要为了“贯彻原则”单独发起无关重构；但当前任务若必须继续扩展该文件，应避免再把新职责直接塞进同一入口，优先复用既有共享模块，或做最小必要的职责下沉。
- 新增或修改 `*.hpp` 时，必须把头文件视为局部接口契约，至少补齐模块/类职责说明，以及关键 public API 的输入、输出、使用前提或非目标范围；不要只让行为埋在 `*.cpp` 实现里。
- 如果 public 接口、所有权边界、缓存语义或调用前提发生变化，必须在同一次任务里同步更新对应头文件说明；不要让头文件声明与实现长期漂移。

## 明确禁止项

- 未经用户明确要求，不得新增 fallback、compat layer、adapter、shim、feature flag、双写路径或新旧逻辑并存。
- 不得为了“更稳妥”新增冗余默认分支、吞错分支、静默降级、重试或猜测性修复。
- 不得绕过现有共享入口直接读取资源、复制语言知识或在局部写死规则。
- 不得因为单个测试失败就修改业务逻辑以迁就偶发时序问题；先复现、重跑并定位根因。
- 不得把当前任务扩展成跨模块重构、命名清理、代码风格统一或目录整理，除非用户明确要求。

## 必须停下来确认的情况

- 需要修改模块边界、单一事实来源或关键共享入口时。
- 需要改变公开行为，尤其是 hover、completion、signature help、diagnostics、semantic tokens 的对外表现时。
- 需要新增资源 bundle、修改资源路径/命名/加载规则，或重新引入当前文档已否定的旧布局时。
- 需要新增跨文件兼容层、迁移层或保留旧行为兜底时。
- 无法用现有架构完成任务，且唯一方案是扩大改动面时。

出现以上任一情况时，先说明触发点、风险和最小可选方案，再等待确认；不要直接实现。

## 实施与验证要求

- 对复杂、多文件、跨模块或高风险改动，修改前先用 3-5 句话说明根因判断和将要采用的最小方案。
- 对低风险、小范围改动，只需一句简短同步，或直接实现后在最终说明中补充根因与方案。
- 只运行与改动范围匹配的最小必要验证；不要默认上全量门禁。
- 如果验证失败，先区分真实回归、环境问题和偶发失败，再决定是否继续改代码。
- 涉及文档更新规则中的任一触发项时，必须在同一次任务里同步更新对应 Markdown 文档。

## 测试踩坑收敛规则

- 对集成测试里已经复现并确认非偶发的坑点，不允许继续保留为“口头经验 / 临时记忆”。
- 处理方式必须在同一次任务里二选一落地：
  - 如果坑点来自测试前提、时序或调用方式，应收敛为共享 helper、统一测试写法，或写入 `docs/testing.md`
  - 如果坑点来自 AI 协作流程、验证习惯或仓库约束，应额外写入 `AGENTS.md`
- 当同类坑点已影响 2 条及以上测试，默认视为“需要制度化”的问题；不要只修当前一条断言然后结束。
- 不得把确认过的测试前提继续散落在单个 test case 里重复手写；优先抽成 shared helper 或明确文档规则。

## 规则清理规则

- 新规则、测试注意事项、AI 约束一旦依赖某个具体架构前提、缓存语义、请求顺序或 helper 约定，就必须把该前提写清楚；不要留下脱离上下文也像“永真”的规则。
- 当架构、共享入口、测试入口或请求链路发生变化时，必须同步检查并清理可能过期的规则、注意事项、helper 注释和文档示例。
- 如果某条规则已经被实现吸收、被共享 helper 消解，或其前提已不再成立，应在同一次任务里删除、改写或降级说明；不要让过期规则继续留在文档里误导后续任务。
- 对已确认过期但本次无法安全清理的规则，必须在最终说明里单独点名，并说明阻塞原因；不能静默保留。

## 最终汇报要求

- 最终说明必须明确包含：根因、实际改动、为何符合当前架构、实际运行的验证、文档是否需要更新。
- 如果没有更新文档，必须明确写出 “No doc updates needed” 或中文等价说明，并说明原因。
- 如果选择不加 fallback 或不做额外抽象，可以直接说明这是为了保持单一事实来源和最小改动面。

## 文档更新规则

只要任务改变了以下任一项，就必须在同一次任务里更新对应 Markdown 文档：

- 构建、运行、测试命令
- 目录结构
- 资源 bundle 路径、命名或加载规则
- 模块边界、单一事实来源、关键架构关系
- 影响 hover、completion、signature help、diagnostics、semantic tokens 的公开行为
- AI 协作流程或仓库约束
- 头文件接口契约、模块职责说明或 AI 维护约束

最低需要检查是否要更新的文档：

- `README.md`
- `AGENTS.md`
- `docs/architecture.md`
- `docs/resources.md`
- `docs/testing.md`

如果本次改动不需要改文档，也要在最终说明里明确写出 "No doc updates needed" 或中文等价说明，并给出原因。

## 关闭任务前的检查清单

结束任务前，按顺序检查：

1. 命令是否变化
2. 路径或命名是否变化
3. 架构或单一事实来源是否变化
4. 测试策略是否变化
5. 文档是否已同步

满足其一为 "是" 时，必须更新相应文档后再结束任务。

## 推荐验证顺序

按改动范围选择最小必要验证：

- 资源/schema 变更：`npm run json:validate`
- TypeScript/client 变更：`npm run compile`
- C++ server 变更：`cmake --build .\\server_cpp\\build`
- 默认集成回归：`npm run test:client:repo`
- 发版前全量门禁：`npm run gate:d3`
- 发版打包：`npm run package:vsix`

## Windows 注意事项

- `server_cpp\\build\\nsf_lsp.exe` 可能因为运行中的 VS Code 扩展宿主或测试进程被锁住。
- 如果 C++ 链接失败且提示 `permission denied`，先确认占用进程已退出，再重新构建。
- 对单条集成测试的偶发失败，先重跑一次再判断是否为真实回归；不要直接据此改业务逻辑。
- 在本机执行 Python 3 脚本时，优先使用 `py -3`；不要假设 `python` 一定指向可用的 Python 3。
